## [Unreleased]

### Bug Fixes

- Fixed `#83`: the crafting store now enables the alchemy panel and submits alchemy attempts based on the active recipe system, instead of any alchemy system present elsewhere in the world.
- Fixed `#92`: `learnRecipe()` now returns localization keys instead of hardcoded English strings, and both crafting UIs localize those notifications when shown to the user.
- Fixed `#93`: `evaluateRecipeAccess()` now reuses its first knowledge-mode access evaluation instead of recomputing the same item-matching work for craftability.
- Fixed repeated component compendium imports creating duplicates in the same crafting system. `CraftingSystemManager` now preserves live `sourceUuid` and canonical `sourceItemUuid` separately, treats repeated or moved compendium entries with the same canonical source as the same component, enforces a per-system unique source-reference constraint during create/update/import, and keeps runtime crafting/alchemy matching aligned with the expanded source-reference chain.

### CI/CD

- Added a manual-only `codex-team-b-manual.yml` workflow for explicitly selected Team B issue runs with Codex. The existing Claude scheduled workflows remain unchanged; the Codex path is opt-in via `workflow_dispatch`, requires `OPENAI_API_KEY`, and still runs `npm test` plus `npm run build` before PR creation.
- Added Conventional Commits enforcement: commitlint.config.js configures @commitlint/config-conventional with an issue-scoped format. A new conventional-commits.yml workflow validates all commits on a PR and the PR title itself. Agent config files and team-b-backlog.yml have been updated with explicit commit format rules.
- Added Foundry VTT integration smoke-test harness: docker-compose.foundry.yml launches a disposable felddy/foundryvtt container; four new scripts (foundry-test-up.mjs, foundry-test-run.mjs, foundry-test-down.mjs, foundry-test.mjs) orchestrate start, Playwright smoke test, and teardown. Artifacts (summary.json, screenshots, console log) are written to test-results/.
- Added foundry-integration.yml GitHub Actions workflow that triggers on push to main, weekly schedule, and workflow_dispatch. On failure, opens a GitHub Issue with the foundry-smoke-failure label.
- Added semantic-release pipeline: release.config.js drives version bumping, CHANGELOG updates, module.json version injection, and GitHub Release creation. A new release.yml workflow runs semantic-release on every push to main. scripts/release.js gains a --version flag for version injection.
- Updated team-a-research.yml UX audit job to install Playwright and run the Foundry smoke harness for live screenshot capture before UX review.
- Added .env.foundry to .gitignore to prevent accidental credential commits.

### Fixed

- Managed result items with missing source documents no longer inherit an arbitrary item type from the crafting actor's inventory. `CraftingEngine._createSingleResult()` now falls back to a deterministic `loot` item type and emits a warning when it has to create fallback item data. Added regression coverage for the fallback type behavior. (#91)
- Unknown crafting resolution modes no longer resolve every result group and report a false craft success. `ResolutionModeService.resolveResultGroups()` now returns an error disposition for unknown modes, and `CraftingEngine.craft()` treats that disposition as a craft failure before any result items are created. Added regression coverage for both the fallback and engine failure path. (#98)
- `CraftingEngine._consumeIngredients()` no longer throws when a matched item has no `system` object. Ingredient consumption now defaults missing `system.quantity` to `1`, which keeps crafting functional for system-agnostic or non-standard item documents. Added a regression test covering the missing-`system` path. (#84)
- Saving a new recipe no longer creates a duplicate entry or shows two success notifications. The recipe editor save button was changed from `type="submit"` to `type="button"`, preventing the form from firing both a submit event and the click handler simultaneously. (T-036)
- `Catalyst.validate()` no longer rejects a catalyst with a non-positive or null `maxUses` value when `degradesOnUse` is false. `maxUses` validation is now skipped entirely when `degradesOnUse` is disabled, because the field is irrelevant in that case. (T-051)
- Saving a recipe with "Restrict visibility to specific users" checked but no users selected no longer raises a validation error. An empty `allowedUserIds` list is now a valid configuration meaning the recipe is hidden from all players. (T-052)
- Disabled recipes in the Crafting Admin panel recipe list are no longer unclickable. A CSS class name conflict between the `.disabled` utility class and the recipe row's enabled state caused browser-native pointer-event blocking on the entire row, preventing GMs from opening the recipe editor or toggling the enabled checkbox. The row class has been renamed to `.recipe-disabled` to eliminate the conflict. Disabled recipes now also display a grey "Disabled" badge alongside their name in the list. (T-053)
- The Crafting App no longer fails to open with a Svelte `each_key_duplicate` error when an actor's stored crafting runs contain duplicate run IDs or when the same run ID appears in both the active-runs and run-history lists. `craftingStore.js` now deduplicates both lists via a Set before they reach the component, and `RunSummary.svelte` uses composite keys (`active-${run.id}` / `history-${run.id}`) to prevent cross-list key collisions. (T-168)
- Ingredient labels in the Crafting UI no longer display as "managed item" or "specific item" for component-type ingredients. Catalyst labels in `evaluateCraftability` no longer display as `undefined`. Both now resolve to the actual component name via `RecipeManager.resolveComponentName()`. Recipe icons now use a fallback chain: custom `recipe.img` → linked recipe item image (via `linkedRecipeItemUuid`) → default document icon, resolved via `RecipeManager.resolveRecipeIcon()` (sync) and `RecipeManager.resolveRecipeIconAsync()` (async). (T-090)
- Completed simple crafts (single-step recipes) no longer persist as "in-progress" in the Crafting App after the craft finishes. `CraftingRunManager` now maintains an in-memory cache keyed by actor ID; `_persist()` writes to both the cache and Foundry actor flags simultaneously, so `_getContainer()` always returns the latest state without waiting for the flag propagation round-trip. The new `invalidateCache(actorId)` method lets external code force a cache eviction when needed. (T-091)
- The `/craft` chat command no longer posts two chat messages on a successful or failed craft. A duplicate `ChatMessage.create()` call that had been added to the chat-command handler in `main.js` alongside the engine's own `_postCraftChatMessage()` has been removed. The handler now uses `ui.notifications.info` / `ui.notifications.error` for inline feedback only, and the engine remains the single canonical source of crafting chat output. (T-091)
- Recipe card icons in the Crafting App no longer render oversized when a recipe's linked image is larger than the card layout expects. The `.recipe-icon img` rule now constrains images to a maximum of 64 x 64 px with `object-fit: cover` and prevents flex-container stretching via `flex-shrink: 0`. (T-091)
- Recipe editor validation errors are now visible inline. Field-level error indicators appear on the recipe name input, ingredient group cards, and result group panels when a field fails validation, supplementing the sticky ValidationBanner at the top of the editor. The "scroll to error" action now guards against collapsing already-expanded panels before scrolling. (T-095)
- The Components panel search bar and heading in the recipe editor no longer scroll out of view when the component list is long. The header area is now wrapped in a sticky container (`position: sticky`) and the panel uses `overflow-y: auto` so the search input remains visible while scrolling through results. (T-095)
- Recipe component cards in the recipe editor no longer overflow their container when names are long or the panel is narrow. Cards now use `box-sizing: border-box` and `max-width: 100%` to stay within their column, and card names are line-clamped to two lines to keep card heights consistent. (T-095)
- Resolved all 38 Svelte compiler warnings across the UI component tree. Added `svelte-ignore state_referenced_locally` directives for stable store prop captures, `id`/`for` attribute pairs for form label accessibility, `aria-label` attributes on icon-only buttons, and corrected ARIA roles on interactive containers and drop zones.
- Dropping a compendium item onto the Items tab drop zone in the Recipe Manager no longer silently fails with "Drop an Item document from sidebar or compendium." The `onDropItem` handler in `SvelteRecipeManagerApp` now resolves the UUID using both the `uuid` field (world item drag) and the `{ pack, id }` pair (compendium item drag), matching the resolution strategy already used by the legacy `RecipeEditorApp`. The warning message has also been extracted to the `FABRICATE.Admin.Items.DropInvalidItem` i18n key. (PLAN: Phase 1)
- `getDragEventData` in `foundryBridge.js` now falls back to parsing `text/plain` from the drag event's `dataTransfer` when the Foundry v13 `TextEditor.implementation.getDragEventData` API is unavailable. Previously, any Foundry version or environment where that API path was absent caused every drop on the Recipe Manager Items tab (and all other Svelte drop zones) to return `null` and silently skip the drop callback, producing the Foundry-native "Drop an Item document from sidebar or compendium." error. The same fix is applied to `foundryCompat.js` (legacy Handlebars path), which also switches from eager module-load-time evaluation of the `TextEditorCompat` constant to lazy per-call evaluation so the API path is checked at the moment of the drop rather than at import time. (PLAN: getDragEventData fallback)
- Dropping an Actor, JournalEntry, Scene, or other non-Item document onto the Items tab drop zone in the Crafting Admin panel no longer silently succeeds or produces a confusing error. The drop handler now inspects the `type` field of the drag data before resolving the UUID. Actors are rejected with a warning notification. Folder drops are expanded and each contained Item is imported individually, with a summary notification reporting how many items were added. Documents of any other non-Item type are rejected with a warning naming the document type. `CraftingSystemManager.addItemFromUuid()` also enforces the type guard server-side: if `fromUuid()` resolves to a document whose `documentName` is not `"Item"`, the method throws. Both legacy app classes (`RecipeManagerApp`, `RecipeEditorApp`) have been wrapped in try/catch so a throw from the type guard produces a user-facing notification rather than an uncaught exception.
- `CraftingSystemManager.addItemFromUuid()` now returns `{ item, action }` instead of a bare item object. `action` is `"added"` when a new component was created, `"updated"` when an existing component's metadata (name or image) was refreshed from the source item, or `"skipped"` when the item was already registered and its metadata is already current. The Svelte and legacy drop handlers both use this result to show a notification only when an update actually occurred. `addItemsFromPack()` now returns `{ added, updated, skipped, total }`, extending the previous `{ added, skipped, total }` shape with an `updated` count for items that were re-imported with refreshed metadata. A new `_findExistingBySourceChain()` helper implements two-level matching: an exact `sourceUuid`/`sourceItemUuid` match, and a fallback-chain match where the dropped UUID is found in the item's `fallbackItemIds`. On an exact match where name or image differs, the stored metadata is overwritten. On a fallback-chain match, the old `sourceUuid` is pushed into `fallbackItemIds` before the new UUID is written, preserving the audit trail across compendium moves.

### Changed

- Renamed the `cauldron` resolution mode to `alchemy`. The canonical value for `system.resolutionMode` is now `"alchemy"` everywhere: `CraftingSystemManager._normalizeSystem()`, `ResolutionModeService`, `RecipeVisibilityService`, `CraftingEngine.craftAlchemy()`, and the Crafting App store. Existing crafting systems persisted with `resolutionMode: "cauldron"` are automatically normalised to `"alchemy"` on load; no manual migration is required. `_normalizeAlchemyConfig()` continues to accept `"cauldron"` during the transition window so that any data written before this change is not silently discarded. 3 new regression tests added to `tests/alchemy-mode.test.js`. (T-189)
- The `mapped` and `tiered` resolution modes have been unified into a single `routed` mode with a recipe-level `resultSelection.provider` field. Provider values are `ingredientSet` (formerly mapped: ingredient-set routing via `IngredientSet.resultGroupId`), `macroOutcome` (formerly tiered: crafting check macro returns a named outcome matched against result group names), and `rollTableOutcome` (new: a FoundryVTT roll table draw determines the result group). Outcome matching is case-insensitive with reserved fail/miss keyword handling. Legacy `mapped` and `tiered` recipes are automatically normalised to `routed` on load; no user migration is required. The recipe editor now exposes a provider selector in the result selection panel. 40 new tests. (T-166)
- Favourited recipes are no longer rendered as oversized cards in a dedicated section above the recipe list. The standalone FavouritesSection component has been replaced with a "Favourites only" toggle button in the FilterBar, matching the existing "Craftable only" filter pattern.
