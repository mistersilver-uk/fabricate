# Changelog

All notable changes to Fabricate will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/).

## [Unreleased]

### Fixed

- Saving a new recipe no longer creates a duplicate entry or shows two success notifications. The recipe editor save button was changed from `type="submit"` to `type="button"`, preventing the form from firing both a submit event and the click handler simultaneously. (T-036)
- `Catalyst.validate()` no longer rejects a catalyst with a non-positive or null `maxUses` value when `degradesOnUse` is false. `maxUses` validation is now skipped entirely when `degradesOnUse` is disabled, because the field is irrelevant in that case. (T-051)
- Saving a recipe with "Restrict visibility to specific users" checked but no users selected no longer raises a validation error. An empty `allowedUserIds` list is now a valid configuration meaning the recipe is hidden from all players. (T-052)
- Disabled recipes in the Crafting Admin panel recipe list are no longer unclickable. A CSS class name conflict between the `.disabled` utility class and the recipe row's enabled state caused browser-native pointer-event blocking on the entire row, preventing GMs from opening the recipe editor or toggling the enabled checkbox. The row class has been renamed to `.recipe-disabled` to eliminate the conflict. Disabled recipes now also display a grey "Disabled" badge alongside their name in the list. (T-053)

### Changed

- Catalyst model aligned with spec 002: constructor now accepts only `systemItemId`, `degradesOnUse`, `destroyWhenExhausted`, and `maxUses`. (T-002)
- `Catalyst.applyDegradation()` now reads and writes `Item.flags.fabricate.catalystItemUsage.timesUsed` instead of the legacy `catalystUses` flag. (T-002)
- `Catalyst.toJSON()` and `Catalyst.fromJSON()` now round-trip only the four spec-defined fields; unrecognised legacy fields are silently ignored on read. (T-002)
- `RecipeManager.createRecipe()` and `RecipeManager.updateRecipe()` now emit a `console.debug` line containing the recipe name and ID after each successful persistence operation. (T-036)
- `Catalyst.applyDegradation()` now performs on-read migration of the legacy `catalystUses` bare-number flag to the spec-compliant `catalystItemUsage.timesUsed` shape. The legacy flag is left in place after migration; subsequent calls use the new flag exclusively. (T-006)
- `CraftingEngine` now applies a failure consumption policy when a crafting check fails: by default, ingredients are consumed (`consumeIngredientsOnFail: true`) and catalysts are not degraded (`consumeCatalystsOnFail: false`). Both flags are read from `system.craftingCheck.consumption` and can be overridden per crafting system. Previously, nothing was consumed on check failure. (T-008)
- `CraftingSystemManager._normalizeEssenceDefinition()` now produces `icon` (string, defaults to `fas fa-mortar-pestle`; empty or whitespace values also fall back to the default) and `sourceItemUuid` (string or null, falls back to legacy `associatedSystemItemId` for existing data). Both fields are dual-written: `sourceItemUuid` is authoritative and `associatedSystemItemId` is kept in sync as a transitional alias. (T-003)
- `CraftingSystemManager.deleteItem()` now clears `sourceItemUuid` and `associatedSystemItemId` on any essence definition that referenced the deleted managed item. (T-003)
- `RecipeManagerApp._prepareContext()` now resolves `associatedItemName` for each essence definition using `sourceItemUuid` (falling back to `associatedSystemItemId`) when preparing the template context. (T-003)
- `RecipeManagerApp._onAddSystemEssenceDefinition()` now reads `icon` and `sourceItemUuid` from the add-essence form and writes both fields (plus `associatedSystemItemId` alias) to the new essence definition. (T-003)
- `CraftingEngine._createSingleResult()` now gates `_transferEffects` on both `recipe.transferEffects` and `system.features.effectTransfer`. Previously, effect transfer executed whenever `recipe.transferEffects` was true regardless of system configuration. (T-005)
- `RecipeManagerApp._prepareContext()` now exposes `enableEffectTransfer` in the template context, and the `toggleFeature` action mapping includes `effectTransfer`. (T-005)
- `Catalyst`, `Ingredient`, and `Result` models now use `componentId` as the primary field name for the managed item reference. Constructors accept `componentId` first with `systemItemId` as a read-time fallback. `toJSON()` emits both `componentId` (primary) and `systemItemId` (transitional alias) so existing consumers continue to work. (T-038)
- `Ingredient._normalizeMatch()` now normalises `match.type: "systemItem"` to `"component"` on read, and resolves `match.systemItemId` into `match.componentId`. The canonical match type for managed item references is now `"component"`. (T-038)
- `CraftingSystemManager._normalizeSystem()` now exposes managed items under `system.components` (primary). The `system.managedItems` property is retained as a transitional alias pointing to the same array. (T-038)
- `RecipeManager._getComponent()` and `CraftingEngine._createSingleResult()` now read managed items from `system.components` first, falling back to `system.managedItems` and then `system.items` for compatibility with older data shapes. (T-038)
- UI action names in `RecipeEditorApp` and `RecipeManagerApp` renamed: `deleteSystemItem` â†’ `deleteComponent`, `editSystemItem` â†’ `editComponent`, and corresponding helper methods updated accordingly. (T-038)
- `CraftingSystemManager._normalizeRecipeVisibility()` now accepts and normalises three `listMode` values â€” `"global"`, `"player"`, and `"knowledge"`. Invalid values fall back to `"global"`. The default `listMode` for systems with no explicit value set has changed from `"player"` to `"global"`. Existing systems that had already stored an explicit `listMode` are unaffected. (T-052)
- `RecipeEditorApp._getSystemFeatureState()` now exposes `showRecipeVisibilityGlobal`, `showRecipeVisibilityPlayer`, and `showRecipeVisibilityKnowledge` flags so the recipe editor template can show or hide controls relevant to the active list mode. (T-052)
- **BREAKING** `CraftingEngine._transferEffects()` now implements the spec 005 essence-based effect transfer pipeline. Effects are no longer read from consumed ingredient items directly; instead they are collected from the `sourceItemUuid` of each `EssenceDefinition` whose essence ID contributed to the resolved ingredients. The old ingredient-level `extractEffects` and `effectFilter` fields on `Ingredient` are no longer consulted and have no effect. Effect transfer now requires all three of the following flags to be `true`: `recipe.transferEffects`, `system.features.effectTransfer`, and `system.features.essences`. Previously, two flags were sufficient and essences did not need to be enabled. (T-009)
- Documentation updated to use `Component`/`componentId` terminology exclusively. All references to the legacy `SystemItem`, `systemItemId`, and `associatedSystemItemId` names have been removed from user-facing docs. Remaining migration notes use past tense to indicate the old names are deprecated. (T-039)
- On module initialisation, Fabricate now automatically migrates legacy field names in all stored recipes and crafting systems: `systemItemId` is renamed to `componentId` in catalysts, results, ingredients, and salvage data; `match.type: "systemItem"` is normalised to `match.type: "component"` in ingredient match objects; and the top-level `managedItems` array in crafting systems is renamed to `components`. The migration runs before any managers load data and is idempotent. No user action is required. (T-040)
- README.md rewritten to match current spec: legacy concepts removed, API examples updated to reflect the `componentId` rename and current service API, salvage and time/currency features documented, data model section aligned with spec 002 and 005. (T-027)
- docs/quickstart.md rewritten to cover the full crafting workflow: creating a crafting system in the GM admin panel, adding managed items via drag-and-drop, creating recipes in the recipe editor, and the player crafting flow. The previous macro-only quickstart has been replaced with a UI-first guide. A macro API section is retained for power users.
- `spec/002-data-models.md`, `spec/006-recipe-visibility.md`, and `spec/003-ui-integration.md` now define `recipeVisibility.listMode` as a three-value enum: `"global" | "player" | "knowledge"`. The default when unset or invalid is `"global"`. The listing algorithm in spec 006 includes a first-class `global` branch. UI integration spec 003 scopes per-recipe visibility controls to `player` and `knowledge` modes only. (T-064)
